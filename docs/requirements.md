# 要件定義書

Brewia は、コーヒー豆の出会いと抽出体験を蓄積するパーソナル記録アプリである。本書は、これまでに完了した GitHub Issue・PR・コミット履歴を仕様の情報源とし、現行プロダクトが満たす要件を再構築したものである。

---

## 1. プロダクト概要

### 1.1 ビジョン

世界各国の豆との出会いと、その豆をどう抽出してどう味わったかを、単なる記録ではなく「コーヒー体験の航海日誌」として蓄積する。

### 1.2 主要ユースケース

- 豆を登録する（手入力 / 写真から自動抽出）
- 豆ごとの抽出レシピと結果カップを記録する
- 抽出時に時間をラップ計測し、注湯ステップとして残す
- レシピをプリセット化し、次回の抽出登録で再利用する
- 抽出に味覚評価とフレーバータグを付与する
- 過去の抽出履歴を豆ごとに振り返る

### 1.3 想定ユーザー

- 自宅や趣味でハンドドリップ抽出を行うコーヒー愛好家
- スペシャルティコーヒー豆を継続的に試したいユーザー
- 抽出条件と味の関係を可視化したいユーザー

## 2. 機能要件

各機能の出典 Issue / PR を `(#番号)` で示す。

### 2.1 ユーザー認証 / アクセス制御

- Google OAuth によるサインイン / サインアウト `(#82, #83, #100, #107, #113)`
- セッションは Turso（libSQL/SQLite）に永続化
- `/login` と `/api/auth/*` 以外のページ・API はログイン必須。未認証は `/login` にリダイレクト
- すべての豆 / 抽出 / プリセットはユーザー単位で完全に分離
- 他ユーザーのリソースへのアクセスは 404 を返す（存在を露出させない）

> Email Magic Link は当初実装されていたが `(#82)`、運用簡素化のため廃止し Google OAuth のみに統一された `(#107, #113)`。Apple ログインは未実装の継続課題 `(#106)`。

### 2.2 豆の管理

| サブ機能 | 出典 |
| --- | --- |
| 豆の新規登録 / 編集 / 削除 | #31, #34, #50 |
| 国・地域・農園・品種・精製・焙煎度・ロースター・メモ・価格 (JPY) の保持 | #5, #33, #89 |
| 国は固定リストから選択（19カ国 + Blended） | #87, #90 |
| 焙煎度は 8 段階（Light / Cinnamon / Medium / High / City / Full City / French / Italian） | #58, #72 |
| 焙煎度入力時にカラーパレットを表示 | #58, #72 |
| 豆の写真から各フィールドを LLM で自動抽出 | #32, #74 |
| 豆の写真から焙煎度を CIELAB L\* 曲線で自動推定 | #59, #77 |
| 豆の削除時に紐づく抽出 / フレーバー紐付けも整合的に削除 | #50 |

### 2.4 抽出タイマー

- 抽出登録フォーム上で動作するラップタイマー `(#62, #80)`
- センチ秒（1/100 秒）まで表示
- ラップ時に直近の経過秒を 5 秒単位に丸めて注湯ステップ行へ反映
- Stop / Reset の 2 段階フロー
- Reset 押下時、ステップ行も初期化される旨の確認ダイアログを表示 `(#98, #104)`

### 2.5 抽出プリセット

- 抽出レシピを名前付きプリセットとして保存 / 編集 / 削除 `(#85, #93, #97, #105, #108)`
- プリセット項目: 名前・説明・既定豆量・既定湯温・抽出ステップ配列
- プリセットは `/presets` で一覧表示 / 個別編集
- プリセット保存ダイアログは抽出フォームの `Extraction Steps` ヘッダから起動 `(#108)`
- 固定 / 組み込みプリセットは廃止し、ユーザー作成プリセットのみを表示 `(#97)`
- プリセット編集ダイアログの抽出ステップ入力 UI は新規抽出フォームと同形 `(#109, #115)`
- 比率ベース管理 / 豆量・湯量の双方向自動スケーリングは継続課題 `(#116)`

### 2.6 フレーバー

- フレーバーは全ユーザー共通のマスタとして共有
- 抽出 1 件に複数のフレーバーをタグ付け可能（中間テーブル `brew_flavor`）
- フレーバーマスタは `GET /api/flavors` で取得（読み取り専用）

### 2.7 ホーム / 統計

- ログイン後のホームに「Your Coffee Journey」サマリ（Total Brews / Bean Variety）を表示
- 豆の一覧をカード形式で表示し、空状態のときは登録誘導 CTA を出す
- ヘッダから Presets / 豆登録 / ユーザーメニューへ遷移

### 2.8 PWA

- マニフェスト `/manifest.webmanifest` と Service Worker `/sw.js` を配信 `(#61, #70)`
- ホーム画面追加でスタンドアロン表示
- 静的アセットは Cache First、その他 GET は Network First、`/api/*` は素通し
- ナビゲーション失敗時は `/offline` フォールバック

## 3. 非機能要件

### 3.1 セキュリティ / マルチテナント

- DB アクセスは Repository 層で `WHERE user_id = ?` を必須化し、TypeScript 型で `userId` の渡し忘れをコンパイル時に検出
- Route Handler は `getAuthenticatedUser()` で 401 を返し、Server Component は `requireUser()` で `/login` へリダイレクト
- 他ユーザーリソースは 403 ではなく 404 を返し、存在情報を漏らさない
- `flavor` のみ共有マスタとして `user_id` を持たない

### 3.2 データ整合性

- 豆削除時は紐づく `brew` と `brew_flavor` を Repository のトランザクション内で連鎖削除
- 抽出更新時は `brew_flavor` の中間テーブルを再構成（既存削除 → 再作成）
- スキーマ列はすべて `NOT NULL` + 既定値（空文字 / 0）を採用

### 3.3 入力バリデーション

- すべての Create / Update API は Zod スキーマ `upsertBeanSchema`, `upsertBrewSchema`, `upsertBrewPresetSchema` を通す
- 国 / 焙煎度は `enum` 制約、味覚評価は 0–5 整数、湯温は 0–100 の数値
- 写真アップロードは `image/jpeg|png` のみ、サーバー側上限 4.5 MB

### 3.4 LLM 連携

- 既定モデルは Claude（`claude-3-5-haiku-20241022` / Haiku 4.5 を併用）
- 失敗時は API エラー / パースエラーを区別し、UI には Toast でメッセージを返す
- LLM コール元エンドポイント `/api/beans/extract` も認証必須（コスト発生防止）

### 3.5 動作環境

- Web ブラウザ（モバイル幅 `max-w-md` を基準にしたレイアウト）
- PWA としてホーム画面に追加可能
- 一部の `pnpm build` は Google Fonts の取得ネットワーク状況に依存

## 4. 用語集

| 用語 | 定義 |
| --- | --- |
| Bean | 焙煎済みコーヒー豆の登録レコード |
| Brew | 1 回の抽出ログ。豆量・湯量・ステップ・評価を持つ |
| Brew Step | 注湯ステップ。経過秒 `time` と累計湯量 `water` の組 |
| Brew Preset | 抽出レシピのひな形。名前・既定値・ステップを保持 |
| Flavor | 風味タグ（共有マスタ） |
| BrewFlavor | Brew と Flavor の中間テーブル |
| User | Auth.js が管理する認証ユーザー |

## 5. スコープ外（明示的な Non-goals）

| 項目 | 理由 / 出典 |
| --- | --- |
| ユーザー間データ共有 / 公開 | プライベート記録に限定 |
| チーム / 組織テナンシー | 単一ユーザー前提 |
| ロール（admin 等） | 認可は ownership のみ |
| パスワード認証 | OAuth に限定 `(#107, #113)` |
| Apple / その他 OAuth プロバイダ | 継続課題 `(#106)` |
| プッシュ通知 / バックグラウンド同期 | PWA スコープ外 |
| 動的 API レスポンスの永続キャッシュ | オフライン書き込み非対応 |
| プロフィール編集画面 | 最小表示のみ |
