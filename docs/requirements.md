# 要件定義書

Brewia の機能要件・非機能要件をコミット履歴・closed issue・merged PR から整理したドキュメント。
DB スキーマの正本は [データ仕様書](./data-spec.md)、API の正本は [API 仕様書](./api-spec.md)、画面の正本は [画面仕様書](./screen-spec.md) を参照。

**対象範囲**: `origin/main`（`c4876b6` 時点）相当の実装  
**更新ポリシー**: 新機能追加・仕様変更 PR がマージされた際に更新する。各要件には裏付け issue/PR を `→ #NNN` 形式で明記する。

---

## 1. プロダクトビジョン

Brewia は「豆（Bean）」と「抽出（Brew）」を中心に、コーヒー体験を記録・振り返るアプリである。
単なる記録ツールではなく、豆ごとの履歴と抽出ごとの体験が積み重なっていく **コーヒー体験の航海日誌** として位置づける（`README.md:3`）。

---

## 2. ターゲットユーザー

- スペシャルティコーヒーに関心があり、自宅での抽出を楽しむ個人ユーザー
- 豆の産地・品種・焙煎度を気にし、抽出ごとの風味の変化を追いたいユーザー
- ログを溜めてのちに振り返りたいユーザー（単回記録ではなく継続利用）

---

## 3. 主要ユースケース

`docs/development-guide.md:9-15` の記述と整合する以下の 4 つを基本とする。

1. **豆を登録する** — 産地・品種・焙煎度・ロースターなど豆の属性を登録・編集・削除する
2. **抽出ログを残す** — 豆に紐づく抽出レシピ（豆量・湯量・湯温・注湯ステップ）を記録する
3. **味覚評価と風味タグを付与する** — 香り・酸味・甘さ・ボディ・総合の 5 評価と風味タグを記録する
4. **豆ごとの履歴を振り返る** — 豆詳細画面で紐づく抽出ログの一覧を確認する

## 4. 機能要件

### 4.1 豆（Bean）管理

→ #32, #33, #59, #87

| 要件 | 説明 |
| --- | --- |
| 豆の CRUD | 豆の新規作成・詳細表示・編集・削除が可能 |
| 必須属性 | 豆名・生産国・焙煎度 |
| 任意属性 | 生産地域・農園・精製方法・品種・ロースター・価格（円）・メモ |
| 写真からの自動入力 | Bean フォームにパッケージ写真を撮影/選択すると Claude Vision API が各フィールドを自動入力する |
| 焙煎度の色推定 | パッケージ画像の色から CIELAB L\* を用いて焙煎度（roast）を推定する |
| コピー作成 | 既存の豆をコピーして新規作成できる |

### 4.2 抽出ログ（Brew）管理

→ #62, #63, #78, #84

| 要件 | 説明 |
| --- | --- |
| 抽出の CRUD | 抽出ログの新規作成・詳細表示・編集・削除が可能 |
| 必須属性 | 対象豆・豆量・湯量 |
| 任意属性 | 挽き目・湯温・注湯ステップ（JSON 配列）・香り/酸味/甘さ/ボディ/総合の 5 評価・風味タグ・メモ |
| 注湯ステップ | `{ time: number, water: number }` の配列で注湯のタイムライン（累計）を記録する |
| 味覚 5 評価 | 香り・酸味・甘さ・ボディ・総合を 0〜5 の整数で記録する |
| 風味タグ | 共有マスタ（`flavor` テーブル）から複数選択・付与できる |
| beanId 絞り込み | `GET /api/brews?beanId=<id>` で特定の豆に紐づく抽出のみ取得できる |
| コピー作成 | 既存の抽出をコピーして新規作成できる |
| 注湯ステップの D&D 並び替え | 抽出フォームで注湯ステップをドラッグ＆ドロップで並び替えできる |

### 4.3 抽出プリセット（Brew Preset）管理

→ #85, #109, #115

| 要件 | 説明 |
| --- | --- |
| プリセットの CRUD | プリセットの新規保存・一覧表示・編集・削除が可能 |
| 属性 | プリセット名・説明・デフォルト豆量・デフォルト湯温・注湯ステップ |
| 新規抽出フォームへの適用 | 新規抽出フォームのプリセット選択でステップ・重量・温度を自動入力する |
| 編集ダイアログ | `/presets/[id]/edit` ページにプリセット専用の編集フォーム（`PresetEditForm`）がある |

### 4.4 写真からの自動入力

→ #58, #61, PR #74

| 要件 | 説明 |
| --- | --- |
| エンドポイント | `POST /api/beans/extract` で multipart/form-data の画像ファイルを受け取る |
| LLM | Anthropic Claude Vision API（`claude-haiku-4-5` デフォルト。環境変数 `ANTHROPIC_MODEL` で上書き可） |
| 抽出フィールド | 豆名・ロースター・生産国・生産地域・農園・品種・精製方法・テイスティングノート |
| 制約 | JPEG/PNG のみ対応、サーバー側上限 4.5 MB |

詳細アーキテクチャは `docs/photo-form-extraction.md` を参照。

### 4.5 抽出タイマー

→ #78

新規抽出フォームにタイマー機能があり、注湯タイミングを計りながらステップ入力できる。

### 4.6 風味マスタ（Flavor）

→ #63

`flavor` テーブルは全ユーザー共有のマスタ。`GET /api/flavors` で一覧取得し、抽出フォームで複数選択できる。`user_id` を持たない（共有マスタのため）。

### 4.7 認証

→ #82, #107, PR #113

現行は **Google OAuth のみ**。メールマジックリンク認証は過去に検討（`docs/auth-architecture.md:31` に記述が残る）されたが、issue #107 / PR #113 で廃止済みであり、現行コードでは実装されていない。

詳細は `docs/auth-architecture.md` を参照。

### 4.8 ユーザーごとのデータ分離

→ #82

- `bean.user_id`・`brew.user_id`・`brew_preset.user_id` は `user.id` に NOT NULL で参照する
- `flavor` は共有マスタで `user_id` を持たない
- middleware により `/login`・`/offline`・`/api/auth/*` 以外は全て要認証
- 他ユーザーのリソースへのアクセスは 404 を返す（403 はリソースの存在を漏らすため）

## 5. 非機能要件

| 区分 | 要件 |
| --- | --- |
| PWA | ホーム画面への追加でインストール可能（スタンドアロン表示）。`/manifest.webmanifest` と `/sw.js` を配信する |
| オフラインフォールバック | ネットワーク不通時は `/offline` ページにフォールバックする（`public/sw.js` の Network First 戦略） |
| 認証必須化 | `/login`・`/offline`・`/api/auth/*` 以外はすべてログイン必須（`middleware.ts` で強制） |
| DB 永続化 | Turso（libSQL/SQLite）に永続化する。Drizzle ORM 経由でアクセスする |
| データスコープ | `bean`・`brew`・`brew_preset` は `user_id` で分離。Repository 層の全クエリに `WHERE user_id = ?` を適用 |
| セッション管理 | `session.strategy = 'database'`（Turso にセッションを永続化） |
| 認証アダプタ | `@auth/drizzle-adapter` を採用 |
| Vercel Preview | `AUTH_REDIRECT_PROXY_URL` 環境変数で Vercel Preview 向けリダイレクトプロキシを設定可能 |
| 対応ブラウザ | モダンブラウザ（Chrome・Safari・Firefox 最新版）。iOS Safari でのホーム画面追加を動作確認対象とする |
| LLM コスト管理 | `POST /api/beans/extract` は認証必須（LLM API コールのコスト管理のため） |

## 6. スコープ外（Non-goals）

`docs/auth-architecture.md:18-25` の Non-goals を引用する。

- ユーザー間のデータシェア機能
- チーム/組織単位のテナンシー
- ロール（admin 等）
- プロフィール編集画面（最小表示のみ将来スコープ）
- パスワード認証
- 複数アカウントの統合/マージ
- プッシュ通知・バックグラウンド同期
- 動的 API レスポンスの永続キャッシュ
