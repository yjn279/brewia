# Brewia 要件定義書

Brewia は、コーヒー豆との出会いと抽出体験を記録するパーソナル・コーヒーログアプリケーションである。豆の産地・焙煎度・プロセスといったスペック情報を管理しながら、その豆でどのように抽出してどのような味わいが得られたかをセットで積み上げていく「コーヒー体験の航海日誌」として機能する。

## ドメイン概要

Brewia が扱うドメインの中心は、コーヒー豆（Bean）と抽出記録（Brew）の 2 エンティティである。豆は産地国・地域・農園・品種・精製処理・焙煎度・ロースターといった属性を持ち、1 つの豆に対して複数の抽出記録が紐づく。抽出記録は豆量・湯量・湯温・挽き目・ポアプロファイル（段階的な注湯のタイムライン）・テイスティング評価（アロマ・酸味・甘味・ボディ・総合）・風味タグ（Flavor）を保持する。

風味タグはマスターデータとして別テーブル（flavor）に保存されており、抽出記録との多対多関係を中間テーブル（brew_flavor）が担う。ユーザーは抽出ごとに複数の風味タグを付けてテイスティングノートを充実させることができる。

## 想定ユーザー

Brewia を使うのは、自宅や職場でハンドドリップやエスプレッソなどのコーヒー抽出を楽しむ個人ユーザーである。複数の豆を試しながら産地や焙煎度の違いを探求する人、または特定の豆で最適なレシピを追求する人が主な利用者層として想定される。マルチユーザー認証機能は現状実装されておらず、単一ユーザーの利用を前提とした設計になっている。

## 機能要件

Brewia が提供する機能は大きく 5 つの領域に整理できる。

豆の管理では、コーヒー豆を登録・編集・削除できる。登録時には豆名・ロースター・産地国・地域・農園・品種・精製処理・焙煎度・メモを入力する。豆名とロースターは必須フィールドで、産地国と焙煎度は選択肢から選ぶ列挙値フィールドである。既存の豆情報をコピーして新規登録する機能（コピー作成）も備えている。

抽出ログの管理では、特定の豆に対する抽出記録を登録・編集・削除できる。各抽出では豆量・湯量・湯温・挽き目（クリック数）・ポアプロファイル（注湯ステップのリスト）・テイスティング評価（各指標 0–5 の整数）・メモを記録する。既存の抽出をコピーして新規登録する機能も備えている。豆を削除すると、紐づく抽出記録と風味タグ中間データも連鎖削除される。

テイスティングノートでは、抽出記録ごとにアロマ・酸味・甘味・ボディ・総合の 5 軸評価（各 0–5 の整数）を付けられる。加えて風味タグをマスターデータから複数選択して付与できる。抽出詳細画面では TasteRadar コンポーネントによるレーダーチャートと PourChart コンポーネントによるポアプロファイルグラフを確認できる。

写真からの自動抽出では、コーヒー豆のパッケージ写真を JPEG または PNG 形式でアップロードすると、LLM（外部 AI API）が画像を解析して豆名・産地・焙煎度などのフィールドを自動入力候補として返す機能を持つ。ファイルサイズの上限はサーバー側で 4.5 MB に設定されており、抽出失敗時は専用のエラーコードで通知される。

PWA としてのインストールおよびオフライン表示では、Brewia はプログレッシブウェブアプリ（PWA）として動作し、ブラウザのホーム画面追加機能でインストールできる。Service Worker（`/sw.js`）が登録され、ネットワーク切断時はオフラインフォールバックページ（`/offline`）に誘導される。

## 非機能要件

技術スタックは Next.js App Router（TypeScript）を採用しており、サーバーコンポーネントと Route Handler を組み合わせた設計になっている。データ永続化は Drizzle ORM 経由で Turso/SQLite に行い、API 入力バリデーションは Zod スキーマを使用している。UI は React + Tailwind CSS で構築されており、コンポーネント設計は Clean Architecture 寄りのレイヤー分離（Route → Service → Repository）に従っている。

PWA 対応の制約として、Service Worker のスコープはルート（`/`）全体に設定されており、オフライン時は `/offline` ページをフォールバックとして表示する。マニフェストは `/manifest.webmanifest` として配信される。モバイルでのズームを防ぐため、ビューポートの `userScalable` は `false` に固定されている。

## ユースケース

豆を新規登録するシナリオでは、ユーザーはホーム画面の「+」ボタンをタップして `/new?type=bean` に遷移し、フォームに豆名・ロースター・産地国・焙煎度など必要情報を入力して保存する。保存後は豆詳細ページ（`/beans/{id}`）に遷移する。

写真から豆情報を自動抽出するシナリオでは、新規豆登録フォーム上で豆のパッケージ写真を選択してアップロードすると、`POST /api/beans/extract` に画像が送信され、LLM による解析結果がフォームに自動入力される。ユーザーはその内容を確認・修正してから保存する。

豆ページから新しい抽出を記録するシナリオでは、ユーザーは豆詳細ページ（`/beans/{id}`）の「+」ボタンをタップして `/new?type=brew&bean={id}` に遷移し、抽出パラメータ・テイスティング評価・風味タグを入力して保存する。保存後は抽出詳細ページ（`/brews/{id}`）に遷移する。

抽出に風味タグを付けるシナリオでは、新規抽出フォームまたは抽出編集ページでフレーバーリストからタグを複数選択する。選択した風味タグは `flavorIds` フィールドとして送信され、`brew_flavor` テーブルに保存される。

オフライン時にフォールバックページを見るシナリオでは、ネットワーク接続が切れた状態でアプリを開くと Service Worker がリクエストをインターセプトし、`/offline` ページを返す。ユーザーは「Back to Home」リンクからオンライン復帰後にホームへ戻れる。
